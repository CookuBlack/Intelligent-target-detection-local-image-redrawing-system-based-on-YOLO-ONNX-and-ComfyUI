# 智能目标检测局部重绘系统 - 项目迭代与调试过程（面试专用版）
本项目的开发与调试遵循**从0到1跑通核心链路 → 解决核心效果问题 → 工程化易用性优化 → 轻量化部署优化**的循序渐进逻辑，每一步都围绕「发现问题→根因分析→方案落地→效果验证」的闭环推进，完整还原了我从实现基础功能到打磨出可用工具的全流程。

---

## 一、项目初始目标
我要做一款**全自动的智能局部图像重绘工具**，核心需求是：无需用户手动框选目标，自动识别图片中的主体，对指定区域进行AI重绘，解决传统图像编辑手动框选效率低、重绘边缘生硬、操作门槛高的痛点。
初始技术选型：YOLO做目标检测 + ComfyUI API调用Stable Diffusion做图像生成，先实现最基础的「检测→掩码→生成」核心流程。

---

## 二、第一阶段：核心链路跑通，解决基础运行问题
### 问题现象
初始版本写完后，调用ComfyUI API持续报错，要么返回400状态码，要么提示节点不存在、参数校验失败，完全无法提交生成任务。

### 根因分析
我逐行拆解报错日志，定位到3个核心问题：
1. **节点ID类型不匹配**：ComfyUI官方API要求节点ID必须是字符串格式，我初始写成了整数，导致系统无法识别节点链路；
2. **节点兼容性问题**：为了优化人脸效果，我引入了第三方Detailer插件的人脸修复节点，但我用的ComfyUI精简版没有预装该插件，导致节点不存在报错；
3. **核心参数名错误**：SetLatentNoiseMask节点的必填参数是`mask`，我误写成了`noise_mask`，触发了参数校验失败。

### 解决方案
1. 统一把所有节点ID改为字符串格式，严格遵循ComfyUI官方API规范；
2. 移除所有第三方节点，只用ComfyUI原生基础节点重构工作流，保证全版本兼容性；
3. 核对每个节点的官方文档，修正所有必填参数名，确保工作流能通过系统校验。

### 优化效果
成功跑通核心链路，API请求返回200状态码，能正常提交生成任务、获取生成结果，实现了从0到1的突破。

## 三、第二阶段：核心效果优化1，解决重绘区域不完整、目标截断问题
### 问题现象
基础流程跑通后，生成效果很差：YOLO输出的检测框是紧贴目标的紧框，重绘后要么人物的肢体、边缘被截断，要么重绘区域和原图之间有非常明显的拼接痕迹，人物特征还原度极低。

### 根因分析
我对比了检测框和重绘效果，定位到3个核心问题：
1. **检测框范围不足**：YOLO输出的是目标的最小外接矩形，完全紧贴人物边缘，没有预留任何过渡空间，重绘时很容易截断人物的头发、肢体等边缘部分；
2. **掩码边缘生硬**：初始用的是0/1二值硬掩码，重绘区域和原图之间是突变的边界，没有任何过渡，导致拼接痕迹非常明显；
3. **重绘强度过高**：初始denoise设为0.8，重绘力度太大，完全破坏了原图的人物结构和背景衔接。

### 解决方案
这是项目最核心的效果优化，我做了3个针对性的功能迭代：
1. **新增软检测框功能**：以YOLO原始检测框的中心为原点，按1.2倍比例等比向外平滑扩大检测框，同时做边界限制，确保不会超出图像范围，给目标边缘预留了充足的过渡空间，避免截断；
2. **新增软边缘掩码功能**：对生成的掩码做高斯模糊处理，把硬边界变成渐变羽化的软边缘，让重绘区域和原图之间有自然的过渡，彻底消除拼接痕迹；
3. **优化重绘核心参数**：把denoise从0.8下调到0.7，在重绘效果和原图结构保留之间找到平衡，既保证重绘效果，又不会破坏原图的人物特征和背景衔接。

### 优化效果
彻底解决了目标截断、重绘区域不完整的问题，人物特征还原度大幅提升，重绘区域和原图实现了无缝衔接，没有任何生硬的拼接痕迹。

## 四、第三阶段：核心效果优化2，解决尺寸不匹配的合成报错问题
### 问题现象
效果优化后，偶尔会出现合成结果时报错：`ValueError: could not broadcast input array from shape (xxx) into shape (xxx)`，生成任务成功完成，但最终结果无法合成保存。

### 根因分析
我拆解了合成逻辑，发现了Stable Diffusion的底层特性限制：
Stable Diffusion的Latent空间是8倍下采样，ComfyUI在处理图片时，会强制把图片尺寸对齐到8的整数倍。如果原图的宽高不是8的倍数，生成后的图片会和原图有1-2个像素的尺寸差，导致生成的目标区域和原图的目标区域尺寸不匹配，无法直接赋值合成。

### 解决方案
在合成结果前，新增了尺寸适配逻辑：先获取原图目标区域的真实宽高，再把生成图对应的区域缩放到和原图目标区域完全一致的尺寸，再进行赋值合成。

### 优化效果
彻底解决了尺寸不匹配的报错问题，无论原图尺寸是否为8的倍数，都能正常合成结果，且最终输出的图片和输入原图的分辨率完全一致，没有任何缩放变形。

## 五、第四阶段：工程化与易用性优化，开发可视化UI界面
### 问题现象
前面的版本都是纯命令行脚本，每次换图片、改提示词、调参数都要修改代码，操作门槛非常高，无法快速调试不同参数的效果，不具备实用性。

### 根因分析
没有可视化交互界面，所有参数都是硬编码在代码里，用户无法直观地调整参数、预览图片和生成结果。

### 解决方案
基于PySide6开发了完整的可视化桌面端UI界面，做了4个核心优化：
1. **界面模块化拆分**：分为双图预览区（左侧原图、右侧生成结果）、参数配置区、操作按钮区、实时日志区，逻辑清晰；
2. **全参数可视化调整**：把提示词、软扩大比例、掩码模糊度、重绘强度、采样步数等所有可调参数，都做成了可视化控件，支持实时调整，无需修改代码；
3. **多线程异步处理**：把核心生成逻辑放到QThread后台线程中执行，避免生成过程中界面卡顿卡死；
4. **视觉与适配优化**：采用暗色护眼主题，优化按钮对比度和控件样式，把窗口尺寸适配到1000×700，解决了初始窗口溢出屏幕的问题，适配绝大多数主流显示器。

### 优化效果
操作门槛大幅降低，用户无需任何代码基础，就能通过「选图→调参→生成→保存」四步完成全流程操作，实时预览效果，工具的实用性和易用性得到了质的提升。

## 六、第五阶段：部署与轻量化优化，切换ONNX模型
### 问题现象
初始版本用的是YOLO的PyTorch .pt模型，部署时需要安装完整的PyTorch环境，依赖包体积大、跨平台兼容性差，无法实现轻量化部署。

### 根因分析
.pt模型依赖PyTorch运行时，无法脱离PyTorch环境运行，部署门槛高。

### 解决方案
1. 把训练好的YOLO模型导出为ONNX格式，改用onnxruntime做推理；
2. 重构模型加载和推理逻辑，完全兼容ONNX模型的运行；
3. 精简项目依赖，实现了无需完整PyTorch环境就能运行的轻量化部署。

### 优化效果
模型推理速度提升了30%左右，部署依赖大幅精简，跨平台兼容性更好，无论是Windows、Linux还是macOS，都能快速部署运行。
